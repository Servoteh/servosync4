1.5

Međukorak — 1.0 sa clouda na naš server

Odobren 4.7.2026

Infrastrukturni prelaz, ne prevođenje koda: PostgreSQL + PostgREST + GoTrue self-hosted na Ubuntu serveru. Front menja samo API URL i ključeve (sve ide kroz jedan config); RLS politike nastavljaju da čuvaju bezbednost nepromenjene.

Uslovi prelaza (faza 1)

  • Pokriti i storage, 12 edge fn i pg_cron ekvivalente
  • Pinned verzije PostgREST/GoTrue — bez auto-update
  • Odlučeno 4.7.: Cloudflare Tunnel — ista javna adresa kao danas, bez VPN-a na uređajima
  • Sunset kriterijum po komponenti (gasi je 3.0)

Šta time dobijamo

  • Puna nezavisnost od Supabase clouda za 2–3 nedelje

  • Strangler-fig u 3.0 ide nad istom bazom
2.0

Proizvodni core iz QBigTehn-a — ovaj repo

U toku

1:1 (minimum) prerada Negovanovog QBigTehn sistema (Access + MS SQL) na NestJS 11 + Prisma 6 + PostgreSQL 16, on-prem. Ne kopira se ceo legacy: ~44% je realno jezgro, 26% je bloat koji ostaje van scope-a.

Baza (on-prem PG 16)

  • ~90 Prisma modela — 1:1 plosnat port legacy šeme
  • Legacy ključ = id (Int), ne UUID · role/statusi String
  • App-owned: users, refresh_tokens, audit_log, bb_sync_*
  • Sync tabele su cache — piše ih samo sync modul

Kod (backend/src)

  • Sync: generički map-driven syncer, 62 entiteta (~520K redova, MSSQL strogo read-only)
  • Auth: JWT login + guard na sync endpointima; fali refresh rotacija i /api/v1
  • Front: login + Sinhronizacije ekrani na ui-kit-u; dizajn v1.0 konačan
  • RBAC/RLS predlog v0.1 napisan — čeka potvrdu

Urađeno

  • Portovana cela šema + rename mapa + migracije
  • Sync svih mapiranih tabela (62) — verifikovan na Dockeru
  • JWT login + guard na sync; front login + Sinhronizacije
  • Mreža: vasa-SQL:5765 dostupan sa ubuntusrv (6.7.)

Sledi (plan A→D u ROADMAP-u)

  • A: PG16 + backend + sync na ubuntusrv; backup; pa CF Tunnel
  • B: refresh rotacija · /api/v1 · no-op RolesGuard + permisije
  • C: pilot TEHNOLOGIJA — TP read-only lista + detalj
  • B4: sastanak Negovan/Nesa — §11 + RBAC (blokira mutacije)

3.0

Objedinjavanje — 1.0 prelazi na stack 2.0

Planirano

Svih ~19 modula 1.0 prelazi na PostgreSQL + NestJS + Next.js i spaja se sa 2.0 u jednu aplikaciju. Strangler-fig, modul po modul, nad istom bazom — bez prekida rada.

Najteži deo — autorizacija

  • ~360+ RLS politika → NestJS guardovi + query-scoping
  • Prvi korak: auth + RBAC paritet (GoTrue → NestJS JWT)
  • RBAC/RLS predlog iz 2.0 je pripremljen baš za ovo
  • Paritet se dokazuje test matricom rola × endpoint

Redosled

  • Auth paritet, pa pilot modul (Lokacije ili Reversi)
  • Front: 83 servisa se repointuju sa PostgREST na NestJS
  • 238 DEFINER RPC → NestJS servisi u transakciji
  • Mobilna (Capacitor ostaje): offline queue, auth/passkeys, push, realtime, SW keš
  • PostgREST/GoTrue se gase kad poslednji modul pređe
4.0

BigBit ERP ulazi u platformu

Planirano

Komercijala (glavna knjiga, PDV/POPDV, fakture, robna dokumenta, magacin, SEF eFaktura) se rebuild-uje u ServoSync domene. BigBit prestaje da bude spoljni izvor istine, bigbit-sync most se ukida. Najveći regulatorni domen — ide tek posle stabilne 3.0.

Materijal već spreman

  • BB_T_25.MDB (297 MB, 201 tabela, ~88k artikala)
  • VBA i forme izvučeni u _analiza/izvoz/

Kad krene

  • Ista metodologija kao za QBigTehn: scope-triage → ekstrakcija logike → mapa šeme
  • Fiskalizacija RS + SEF = posebna analiza

Pravila koja važe kroz sve faze

Jedan izvor istine po tabeli, jednosmerno. „Oba smera" = više jednosmernih tokova; nikad dvosmerno na istim redovima.
Overlay never touch cache. Sync-ovane tabele su cache i piše ih samo sync; lokalna polja idu u overlay/app-owned tabele.
U QBigTehn MSSQL se nikada ne piše. Samo parametrizovani SELECT preko bridge_reader naloga.
Front na Cloudflare, backend samo kroz Cloudflare Tunnel. Ista javna adresa kao postojeća; server se ne izlaže direktno; sync worker zove izvore odlazno.
Svaki privremeni most ima sunset datum. Umire čim se modul integriše u 3.0 — nijedan most nije trajan.
Otvorene odluke niko ne implementira unapred. Dok odluka nije zapisana u BACKEND_RULES, ne postoji.
